# 页面设计分析模块

扫描设计稿/原型中的实际页面，将每个页面的元素布局如实记录为结构化描述。

## 核心原则

**只记录设计稿中实际存在的内容，禁止自行编造任何元素。**

> 这是最重要的原则。页面设计的价值在于忠实还原设计稿的每一个元素。如果设计稿中没有某个元素，就不要写。如果不确定是否存在，标注「[待确认]」而不是猜测。

**三大禁止**：
1. **禁止凭空编造**：不允许自己想象页面元素、搜索条件、表格列、弹窗字段
2. **禁止套用模板**：不允许用通用的列表页/弹窗模板填充，每个页面必须如实反映原稿
3. **禁止概括省略**：不允许把3个搜索条件概括为"若干搜索条件"

**动作判定原则**：原型/设计稿画的是页面当前状态，不代表所有页面都是全新做的。每个页面、每个组件的动作类型（新增/修改/优化）必须根据需求文档中的实际描述来确定：
- 需求明确说是新做的 → 标记为「新增」
- 需求说在已有页面上改 → 标记为「修改」
- 需求说优化已有功能 → 标记为「优化」
- 需求没有明确说 → 标注「[待确认]」

### 子标题命名规则

子标题 = **指令性动作（新增/修改/优化/查看）+ 业务对象 + 元素类型**，动作来自需求文档：

| 正确（指令性标题） | 错误（描述性标题） |
|---------------------|------------------|
| `修改用户列表页` | `用户管理列表页` |
| `修改用户列表搜索条件` | `用户列表搜索条件` |
| `修改用户列表表格列` | `用户列表表格列` |
| `新增用户弹窗` | `弹窗` |

## 分析步骤

1. **逐页扫描**：按设计稿中的页面顺序，一个页面一个页面地看
2. **如实记录**：设计稿中有什么元素就记录什么，没有的就不写
3. **保留原文**：字段名、按钮文字、列名保持设计稿中的原文
4. **标注来源**：每个页面标注来自哪个设计稿文件

## 不同页面类型的记录重点

### 列表页
重点记录：搜索条件（有哪些）→ 表格列（有几列）→ 操作按钮（表格上方的）→ 弹窗（点击按钮后出现的）→ 弹窗字段（弹窗里有几个输入项）

### 表单页
重点记录：表单字段（有哪些）→ 字段类型（输入框还是下拉等）→ 校验规则（必填/格式等）→ 提交按钮

### 详情页
重点记录：信息区块（分几块）→ 展示字段（有哪些）→ 展示方式（文本/标签/图片等）

### 弹窗/抽屉
重点记录：弹窗标题 → 表单字段（有几个、什么类型）→ 操作按钮

## 常见错误（必须避免）

| 错误 | 正确做法 |
|------|----------|
| 设计稿有2个搜索条件，自己编了5个 | 只写设计稿中的2个 |
| 设计稿没有弹窗，自己加了一个"新增弹窗" | 设计稿没有就不写 |
| 把通用的列表页模板直接套用 | 每个页面必须从设计稿中逐个读取元素 |
| 搜索条件写成"若干搜索条件" | 逐个列出每个搜索条件的名称和类型 |
| 弹窗字段写成"包含基本信息字段" | 逐个列出每个字段的名称、控件类型、是否必填 |
| 设计稿中没有标注必填，自己标记了必填 | 没有明确信息的标注「[待确认]」 |
