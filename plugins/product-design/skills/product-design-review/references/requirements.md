# 需求背景分析

分析产品设计的需求背景，理解项目为什么做、解决什么问题。

## 分析维度

### 1. 项目概述
- 项目名称
- 项目类型（新功能/优化/重构）
- 负责团队
- 时间节点

### 2. 业务背景
- 为什么做（业务动机）
- 解决什么问题（痛点）
- 预期收益（业务收益+用户价值）

### 3. 用户需求
- 目标用户是谁
- 核心诉求是什么
- 使用场景有哪些

### 4. 功能范围
- 包含哪些功能
- 不包含什么
- 优先级排序

### 5. 成功指标
- 业务指标
- 用户体验指标
- 技术指标

### 6. 约束与风险
- 时间/资源/技术约束
- 潜在风险

---

## 输出结构

**每个章节必须包含两部分**：
1. **当前设计梳理**：从设计文档中提取的现有设计内容
2. **审查意见**：基于梳理内容给出的问题、风险、建议

## 执行要点

1. **读取设计文档**：使用 Read 工具读取用户提供的 PRD/设计文档
2. **逐页扫描原型/设计稿**：详见下方「原型需求提取方法」
3. **汇总需求**：将文档需求 + 原型中提取的需求统一梳理，填入模板
4. 标注缺失：无法获取的信息，标注 `[待补充]`
5. **追加到主文档**：将分析内容追加到 `产品设计审查报告.md` 的「## 1. 需求背景」章节

**重要**：需求分析是后续所有章节的基础。从原型中提取的需求描述（功能范围、业务规则、页面说明等）会直接影响后续章节的审查内容。后续章节应基于此处提取的需求进行审查，而不是重复提取。

## 原型需求提取方法

原型/设计稿中通常包含大量 PRD 里没有写的需求细节，必须仔细提取。按以下方法逐页扫描：

### 提取什么

原型中的文字不只看表面，每一类文字都是需求信息：

| 文字类型 | 在哪里 | 提取什么 |
|----------|--------|----------|
| 页面标题 | 页面顶部或面包屑 | 这个页面做什么功能 |
| 功能说明/备注 | 页面空白处、侧边备注区 | 功能的业务规则、操作说明 |
| 字段说明 | 输入框上方/下方的小字 | 校验规则、格式要求、默认值 |
| 按钮文案 | 按钮上的文字 | 操作语义（新增/修改/删除/导出） |
| 提示文案 | placeholder、tooltip | 输入引导、格式要求 |
| 状态说明 | 旁注、标注 | 什么时候显示/隐藏/禁用 |
| 流程说明 | 页面间的箭头/连线 | 操作先后顺序、条件分支 |
| 标签/分类 | tab标签、筛选条件 | 数据分类维度、筛选逻辑 |

### 怎么扫描

1. **按页面逐个打开**：不要跳页，每个页面都要看到
2. **先看页面整体**：页面标题是什么，这个页面承担什么功能
3. **再看每个区域**：搜索区、列表区、弹窗区、详情区，逐个看
4. **重点看小字和备注**：原型中不起眼的小字往往是关键的业务规则
5. **记录操作类型**：这个页面/组件是新增还是修改已有功能

### 提取结果记录

将提取到的需求信息记录到需求背景的「功能范围」中，按页面组织：

```markdown
**功能范围**（从原型提取）：

| 页面 | 功能点 | 需求描述（原型中的原文） | 操作类型 |
|------|--------|--------------------------|----------|
| 用户列表页 | 新增筛选条件 | 备注："按注册时间筛选，默认近30天" | 修改 |
| 用户列表页 | 新增导出按钮 | 按钮："导出Excel"，备注："单次最多导出1万条" | 修改 |
| 新增用户弹窗 | 手机号注册 | placeholder："请输入11位手机号" | 新增 |
| 用户详情页 | 全新页面 | 页面标题："用户详情"，展示用户基本信息+订单记录 | 新增 |
```
