# 产品设计审查 Skill

基于**已有设计材料**进行产品设计梳理和审查，输出**可用于下一步工作的设计文档**。

## 核心原则

### 输出定位

**输出的审查报告是一个"可工作的设计文档"**，不是一次性报告——设计师、开发、测试、产品都能直接基于报告工作，不需要再看原始设计稿。

### 输入优先级

```
1. 已有设计文档（设计稿/原型/PRD/产品文档）
2. HTML 文件夹（设计稿导出）
3. 代码库中的 UI 实现
4. 用户口述描述
```

**不要从零开始设计**，基于现有材料进行梳理和审查。

### 多项目处理

提供多个项目时，**逐个执行，禁止并行**。创建 `产品设计审查计划跟踪.md` 跟踪进度，完成后自动继续下一个。详细流程和示例见 [core-principles.md](references/core-principles.md#多项目处理)。

### 保留原始细节

**当前设计梳理**必须保留设计稿原始细节。禁止概括（"页面有3个按钮"）、禁止丢失文案、禁止省略交互细节。示例对比见 [core-principles.md](references/core-principles.md#保留原始细节)。

### 审查意见务实原则

**审查意见必须基于实际设计内容给出，禁止凭空编造问题或建议。** 设计稿里有什么就审什么，没有的不审。不确定的标注「[待确认]」而不是猜测。不要为了凑检查项而杜撰不存在的风险。

**如果本次需求/原型/系统没有涉及某个章节的内容，直接写「不涉及」，不要为了输出而强行填充。**

## 章节选择（必须执行）

**开始审查前，一次性展示所有章节让用户选择。** 用户选完后按顺序依次执行所有选中章节，**中途不再询问是否继续**。

```
请选择要审查的章节（输入编号，多选用逗号分隔，输入 all 全选）：

| 编号 | 章节 | Phase |
|------|------|-------|
| 1  | 需求背景   | Phase 1 |
| 2  | 业务逻辑   | Phase 2 |
| 3  | 场景设计   | Phase 3 |
| 4  | 信息架构   | Phase 4 |
| 5  | 页面设计   | Phase 5 |
| 6  | 交互设计   | Phase 6 |
| 7  | 数据埋点   | Phase 7 |
| 8  | 安全设计   | Phase 8 |

示例：all | 1,2,5 | 5-8
```

用户选择后，按章节编号顺序依次执行，未选中的章节跳过。**禁止在每个 Phase 之间询问"是否继续"**。

## 执行流程

### 收集输入

询问用户：
1. **设计文档路径**：已有的 PRD/设计文档（Word/PDF/MD）位置
2. **设计稿路径**：设计稿/原型导出文件位置
3. **项目名称**：用于输出文档命名
4. **审查范围**：展示章节清单让用户选择

文档读取工具的使用方法见 [document-reading.md](references/document-reading.md)。

### 创建输出文档

收集完输入后、开始分析前，**立即创建输出文档**：

1. 创建目录：`mkdir -p docs/product-design/[人名-编号-项目名]/`
2. 创建文件：`docs/product-design/[人名-编号-项目名]/产品设计审查报告.md`，写入报告头部（标题、时间、来源、审查范围）
3. 后续每个 Phase 的分析结果，使用 **Edit 工具追加**到该文件的对应章节

### Phase 1: 需求背景

1. 读取 [requirements.md](references/requirements.md)
2. 执行需求背景分析
3. 写入报告「1. 需求背景」章节

### Phase 2: 业务逻辑

1. 读取 [business-logic.md](references/business-logic.md)
2. 执行业务逻辑分析
3. 写入报告「2. 业务逻辑」章节

### Phase 3: 场景设计

1. 读取 [scenarios.md](references/scenarios.md)
2. 写入报告「3. 场景设计」章节，用子标题区分：
   - 3.1 使用场景
   - 3.2 新手引导
   - 3.3 边缘场景
   - 3.4 上下文场景

### Phase 4: 信息架构

1. 读取 [information-architecture.md](references/information-architecture.md)
2. 写入报告「4. 信息架构」章节，用子标题区分：
   - 4.1 站点地图
   - 4.2 导航结构
   - 4.3 内容层级

### Phase 5: 页面设计

1. 读取 [page-design.md](references/page-design.md)，**严格按其输出模板格式执行**
2. 结合需求文档确定动作类型，逐页扫描记录
3. 写入报告「5. 页面设计」

### Phase 6: 交互设计

1. 依次读取并写入报告「6. 交互设计」章节，用子标题区分：
   - 6.1 线框图分析 ← [interaction-wireframe.md](references/interaction-wireframe.md)
   - 6.2 用户流程 ← [interaction-flow.md](references/interaction-flow.md)
   - 6.3 微交互 ← [interaction-micro.md](references/interaction-micro.md)
   - 6.4 状态设计 ← [interaction-state.md](references/interaction-state.md)
   - 6.5 提示反馈 ← [usability.md](references/usability.md)

### Phase 7: 数据埋点

1. 读取 [analytics.md](references/analytics.md)
2. 写入报告「7. 数据埋点」

### Phase 8: 安全设计

1. 读取 [security.md](references/security.md)
2. 写入报告「8. 安全设计」

### 附录生成

1. 汇总所有待补充项 → 「附录 A」
2. 输出设计与实现一致性检查 → 「附录 B」
3. 输出风险汇总 → 「附录 C」

## 输出规范

### 单一文档输出

**所有审查内容必须输出到同一个文档，禁止生成多个文件。**

路径：`docs/product-design/[人名-编号-项目名]/产品设计审查报告.md`

示例：
- 有编号时：`docs/product-design/张三-PRD001-用户中心/产品设计审查报告.md`
- 无人名/编号时：`docs/product-design/用户中心/产品设计审查报告.md`

### 文档结构

每个章节包含「当前设计梳理」+「审查意见」+「风险统计」（页面设计模块除外）。完整模板见 [output-template.md](references/output-template.md)

```
# [项目名] 产品设计审查报告
## 1. 需求背景
## 2. 业务逻辑
## 3. 场景设计
  ### 3.1 使用场景 / 3.2 新手引导 / 3.3 边缘场景 / 3.4 上下文场景
## 4. 信息架构
  ### 4.1 站点地图 / 4.2 导航结构 / 4.3 内容层级
## 5. 页面设计
## 6. 交互设计
  ### 6.1 线框图 / 6.2 用户流程 / 6.3 微交互 / 6.4 状态设计 / 6.5 提示反馈
## 7. 数据埋点
## 8. 安全设计
## 附录 A. 待补充项清单
## 附录 B. 设计与实现一致性检查
## 附录 C. 风险汇总
```

**状态标记**：✅ 已设计 / ⚠️ 部分设计 / ❌ 缺失
**风险等级**：🔴 高风险 / 🟡 中风险 / 🟢 低风险
