# 产品设计审查 Skill

基于**已有设计材料**进行产品设计梳理和审查，输出结构化的审查报告。

## ⚠️ 核心原则

### 输入优先级

```
1. 首先阅读已有的设计文档（设计稿/原型/PRD/产品文档）
2. 然后结合代码实现进行验证和补充
3. 输出：对现有设计的"梳理 + 审查 + 补充建议"
```

**不要从零开始设计**，而是基于现有材料进行梳理和审查。

### ⚠️ 多项目处理规范

当用户提供**多个不同项目**的设计文档时：

**禁止行为**：
- ❌ 为了追求速度，并行分析多个项目
- ❌ 胡乱填充数据，把项目A的内容填到项目B
- ❌ 混淆不同项目的设计内容
- ❌ 在一个报告中合并多个项目

**正确行为**：
1. **先列计划**：列出所有待分析的项目清单
2. **逐个分析**：按顺序一个一个项目完整分析
3. **独立报告**：每个项目生成独立的审查报告
4. **确认后继续**：完成一个项目后，确认用户是否继续下一个

**示例对话**：
```
用户: 帮我审查这三个项目的设计：用户中心、订单系统、支付网关

AI: 收到，我将按顺序逐个分析以下项目：

| 序号 | 项目名 | 状态 |
|------|--------|------|
| 1 | 用户中心 | ⏳ 待分析 |
| 2 | 订单系统 | ⏳ 待分析 |
| 3 | 支付网关 | ⏳ 待分析 |

现在开始分析第1个项目「用户中心」，请提供该项目的...
```

### ⚠️ 保留原始细节（重要）

**当前设计梳理**部分必须保留设计稿的原始细节，不要过度概括。

**禁止行为**：
- ❌ 高度概括，如"页面有3个按钮"
- ❌ 丢失具体文案，如"[提交按钮]"而不是"立即注册"
- ❌ 忽略布局细节，如"上方是标题"而不是具体描述位置和样式
- ❌ 省略交互细节，如"有点击效果"而不是"点击后按钮变蓝+显示loading"

**正确行为**：
- ✅ **逐页记录**：每个页面/页面区块都要单独记录
- ✅ **保留原文**：按钮文案、标题、提示语等保持原文
- ✅ **详细描述**：布局结构、元素位置、视觉层级都要描述
- ✅ **记录交互**：点击反馈、状态变化、动画效果都要记录

**示例对比**：

| 错误（过度概括） | 正确（保留细节） |
|------------------|------------------|
| 页面顶部有标题和按钮 | 页面顶部（高度64px）：左侧"用户中心"标题（18px黑色加粗），右侧"退出"按钮（灰色文字链接） |
| 表单有3个字段 | 表单区域：1) 手机号输入框（placeholder: "请输入手机号"，必填，格式校验）；2) 验证码输入框（6位数字，右侧"获取验证码"按钮，60s倒计时）；3) 协议勾选框（默认不勾选，链接"用户协议"） |
| 底部有提交按钮 | 底部固定栏：主按钮"立即注册"（全宽，蓝色#1890ff，圆角4px），禁用态为灰色#d9d9d9 |

## 审查流程

产品设计审查按以下顺序进行，每个阶段可根据需要跳过：

```
1. 需求分析 → 理解项目背景
2. 场景设计 → 分析使用场景
3. 信息架构 → 梳理内容结构
4. 交互设计 → 宣查交互体验
5. 可用性   → 检查提示反馈
```

## 使用方式

### 方式一：完整审查

用户说"帮我审查这个设计"时，按顺序执行所有模块：

1. 先执行需求背景分析
2. 根据分析结果，询问用户是否继续下一阶段
3. 逐步完成所有审查模块

### 方式二：选择性审查

用户指定特定维度时，只执行对应模块：

| 用户说... | 执行模块 |
|-----------|----------|
| "分析需求背景"、"这个设计解决什么问题" | 需求背景 |
| "梳理使用场景"、"用户怎么用" | 使用场景 |
| "新手怎么上手"、"onboarding" | 新手引导 |
| "边缘场景"、"异常情况" | 边缘场景 |
| "不同环境怎么用" | 上下文场景 |
| "站点地图"、"内容结构" | 站点地图 |
| "导航结构"、"导航清晰吗" | 导航结构 |
| "信息层级"、"视觉层级" | 内容层级 |
| "线框图"、"布局分析" | 线框图分析 |
| "用户流程"、"操作路径" | 用户流程 |
| "微交互"、"交互细节" | 微交互 |
| "状态设计"、"状态完整吗" | 状态设计 |
| "提示反馈"、"提示设计" | 提示反馈 |

## 输入处理

### 输入优先级（重要）

```
优先级 1: 已有设计文档（设计稿/原型/PRD/产品文档）
优先级 2: HTML 文件夹（设计稿导出）
优先级 3: 代码库中的 UI 实现
优先级 4: 用户口述描述
```

### 接收输入

用户可能提供：
- **设计稿/原型**（优先）：Figma/Sketch/蓝湖导出的 HTML、图片
- **PRD/产品文档**：Word/PDF/Markdown 格式的需求文档
- **HTML 文件夹**：设计稿导出的完整文件夹
- **代码库**：用于验证设计实现情况
- **纯文本描述**：用户口述的需求和设计思路

### HTML 文件夹扫描（用于验证和补充）

```
/design/xxx/
├── index.html        # 入口文件
├── pages/            # 页面文件夹
├── components/       # 组件文件夹
└── assets/           # 资源文件
```

### 信息提取

| 信息类型 | 提取位置 | 提取方式 |
|----------|----------|----------|
| 页面标题 | `<title>` 或 `<h1>` | 文本内容 |
| 功能模块 | 页面区块划分 | DOM 结构 |
| 交互元素 | 按钮、链接、表单 | 元素识别 |
| 文案内容 | 文本节点 | 文本提取 |

## 输出规范

### ⚠️ 重要：单一文档输出

**所有审查内容必须输出到同一个文档，禁止生成多个文件。**

```
docs/product-design/[项目名]/产品设计审查报告.md
```

**禁止行为**：
- ❌ 每个模块生成独立文件（如 `需求背景.md`、`使用场景.md`）
- ❌ 按 phase 生成多个文档
- ❌ 分散输出到不同路径

**正确行为**：
- ✅ 所有内容追加到 `产品设计审查报告.md` 的对应章节
- ✅ 使用 `##` 标题区分不同模块
- ✅ 每完成一个模块，使用 Edit 工具追加内容到主文档

### 输出路径

所有审查内容输出到**一个结构化的主文档**：

```
docs/product-design/[项目名]/产品设计审查报告.md
```

**核心原则**：每个章节必须包含「当前设计梳理」+「审查意见」两部分

### 状态和风险定义

**状态标记**:
- ✅ 已设计：设计文档和设计稿一致
- ⚠️ 部分设计：有设计但不完整，或设计与实现不一致
- ❌ 缺失：无设计文档且无设计稿

**风险等级**:
- 🔴 高风险：影响用户体验或功能使用
- 🟡 中风险：影响用户满意度
- 🟢 低风险：仅影响一致性或规范性

### 文档格式要求

- 每个 chapter 使用 `##` 标题
- 使用表格展示检查项
- 每个模块输出风险统计
- 最终汇总所有风险项

## 执行流程

### 第一步：收集输入材料

询问用户：
1. **设计文档路径**：已有的 PRD/设计文档（Word/PDF/MD）位置
2. **设计稿路径**：设计稿/原型导出文件位置
3. **项目名称**：用于输出文档命名
4. **审查范围**：完整审查还是选择性审查

### 第二步：读取设计文档

1. 根据文档类型选择合适的读取方式
2. 提取设计文档中的关键信息
3. 记录文档中已有的设计决策

### 第三步：扫描设计稿

1. 分析 HTML/图片等形式的设计稿
2. 发现文档中未提及的设计细节
3. 标记设计与文档不一致的地方

### 第四步：执行分析

根据用户选择，按需读取 references 目录下的模块指南：

| 模块 | 参考文件 | 用途 |
|------|----------|------|
| 需求背景 | [requirements.md](references/requirements.md) | 分析项目背景和需求 |
| 使用场景 | [scenarios.md](references/scenarios.md) | 分析核心使用场景 |
| 新手引导 | [scenarios.md](references/scenarios.md#新手引导) | 分析新用户引导 |
| 边缘场景 | [scenarios.md](references/scenarios.md#边缘场景) | 分析异常情况处理 |
| 上下文场景 | [scenarios.md](references/scenarios.md#上下文场景) | 分析环境适配 |
| 站点地图 | [information-architecture.md](references/information-architecture.md) | 梳理内容结构 |
| 导航结构 | [information-architecture.md](references/information-architecture.md#导航结构) | 分析导航设计 |
| 内容层级 | [information-architecture.md](references/information-architecture.md#内容层级) | 分析信息优先级 |
| 线框图 | [interaction.md](references/interaction.md) | 分析页面布局 |
| 用户流程 | [interaction.md](references/interaction.md#用户流程) | 梳理操作路径 |
| 微交互 | [interaction.md](references/interaction.md#微交互) | 分析交互细节 |
| 状态设计 | [interaction.md](references/interaction.md#状态设计) | 检查状态覆盖 |
| 提示反馈 | [usability.md](references/usability.md) | 检查提示机制 |

### 第五步：输出文档

1. 创建 `docs/product-design/[项目名]/` 目录
2. 将所有分析结果写入 `产品设计审查报告.md`
3. 标注 `[待补充]` 的信息
4. 在附录中列出待补充项和一致性检查结果

## 完整审查流程

当用户选择完整审查时，按以下顺序执行，**所有内容写入同一个文档**：

### Phase 1: 需求分析
1. 读取 [requirements.md](references/requirements.md)
2. 执行需求背景分析
3. 写入报告「1. 需求背景」章节
4. 询问用户是否继续

### Phase 2: 场景设计
1. 读取 [scenarios.md](references/scenarios.md)
2. 依次执行并写入报告：
   - 使用场景分析 → 「2. 使用场景」
   - 新手引导分析 → 「3. 新手引导」
   - 边缘场景分析 → 「4. 边缘场景」
   - 上下文场景分析 → 「5. 上下文场景」
3. 询问用户是否继续

### Phase 3: 信息架构
1. 读取 [information-architecture.md](references/information-architecture.md)
2. 依次执行并写入报告：
   - 站点地图分析 → 「6. 站点地图」
   - 导航结构分析 → 「7. 导航结构」
   - 内容层级分析 → 「8. 内容层级」
3. 询问用户是否继续

### Phase 4: 交互设计
1. 读取 [interaction.md](references/interaction.md)
2. 依次执行并写入报告：
   - 线框图分析 → 「9. 线框图分析」
   - 用户流程分析 → 「10. 用户流程」
   - 微交互分析 → 「11. 微交互」
   - 状态设计分析 → 「12. 状态设计」
3. 询问用户是否继续

### Phase 5: 可用性
1. 读取 [usability.md](references/usability.md)
2. 执行提示反馈分析
3. 写入报告「13. 提示反馈」

### Phase 6: 附录生成
1. 汇总所有待补充项 → 「附录 A」
2. 输出设计与实现一致性检查 → 「附录 B」
3. 完成报告
