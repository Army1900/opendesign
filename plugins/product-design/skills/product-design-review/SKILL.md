# 产品设计审查 Skill

基于**已有设计材料**进行产品设计梳理和审查，输出**可用于下一步工作的设计文档**。

## ⚠️ 核心原则

### 输出定位

**输出的审查报告是一个"可工作的设计文档"**，不是一次性报告：

| 角色 | 用途 |
|------|------|
| **设计师** | 基于报告进行设计修改，不需要再看原始设计稿 |
| **开发** | 基于报告进行开发，了解所有交互细节和业务逻辑 |
| **测试** | 基于报告编写测试用例，了解所有场景和边界条件 |
| **产品** | 基于报告进行评审，补充遗漏的业务需求 |

### 输入优先级

```
1. 首先阅读已有的设计文档（设计稿/原型/PRD/产品文档）
2. 然后结合代码实现进行验证和补充
3. 输出：对现有设计的"梳理 + 审查 + 补充建议"
```

**不要从零开始设计**，而是基于现有材料进行梳理和审查。

### ⚠️ 多项目处理规范

当用户提供**多个不同项目**的设计文档时：

**禁止行为**：
- ❌ 为了追求速度，并行分析多个项目
- ❌ 胡乱填充数据，把项目A的内容填到项目B
- ❌ 混淆不同项目的设计内容
- ❌ 在一个报告中合并多个项目

**正确行为**：
1. **创建计划文档**：在当前执行目录创建 `产品设计审查计划跟踪.md`
2. **逐个执行**：按顺序一个一个项目完整分析
3. **更新计划文档**：完成后立即更新计划文档中的状态
4. **自动继续**：无需等待确认，直接开始下一个项目

**计划文档模板**：见本目录下 `产品设计审查计划跟踪.md`

**示例流程**：
```
用户: 帮我审查这三个项目的设计：张三-PRD001-用户中心、订单系统、支付网关

AI: 收到，我将按顺序逐个分析以下项目：

[创建/更新 计划跟踪文档 `产品设计审查计划跟踪.md`]
| 序号 | 项目标识 | 状态 |
|------|----------|------|
| 1 | 张三-PRD001-用户中心 | ⏳ 待开始 |
| 2 | 订单系统 | ⏳ 待开始 |
| 3 | 支付网关 | ⏳ 待开始 |

[更新状态为🔄进行中，开始分析第1个项目「用户中心」...

[分析完成后, 更新计划文档标记✅已完成]
[自动继续第2个项目「订单系统」...
```

### ⚠️ 保留原始细节（重要）

**当前设计梳理**部分必须保留设计稿的原始细节，这是报告的核心价值：

> 📌 **目标**：看报告的人不需要再看原始设计稿，所有关键信息都在报告里。

**禁止行为**：
- ❌ 高度概括，如"页面有3个按钮"
- ❌ 丢失具体文案，如"[提交按钮]"而不是"立即注册"
- ❌ 忽略布局细节，如"上方是标题"而不是具体描述位置和样式
- ❌ 省略交互细节，如"有点击效果"而不是"点击后按钮变蓝+显示loading"

**正确行为**：
- ✅ **逐页记录**：每个页面/页面区块都要单独记录
- ✅ **保留原文**：按钮文案、标题、提示语等保持原文
- ✅ **详细描述**：布局结构、元素位置、视觉层级都要描述
- ✅ **记录交互**：点击反馈、状态变化、动画效果都要记录

**示例对比**：

| 错误（过度概括） | 正确（保留细节） |
|------------------|------------------|
| 页面顶部有标题和按钮 | 页面顶部（高度64px）：左侧"用户中心"标题（18px黑色加粗），右侧"退出"按钮（灰色文字链接） |
| 表单有3个字段 | 表单区域：1) 手机号输入框（placeholder: "请输入手机号"，必填，格式校验）；2) 验证码输入框（6位数字，右侧"获取验证码"按钮，60s倒计时）；3) 协议勾选框（默认不勾选，链接"用户协议"） |
| 底部有提交按钮 | 底部固定栏：主按钮"立即注册"（全宽，蓝色#1890ff，圆角4px），禁用态为灰色#d9d9d9 |

## 审查流程

产品设计审查按以下顺序进行，每个阶段可根据需要跳过：

```
1. 需求分析 → 理解项目背景
2. 业务逻辑 → 梳理业务规则和流程
3. 场景设计 → 分析使用场景
4. 信息架构 → 梳理内容结构
5. 页面设计 → 扫描页面元素布局 ⭐ 新增
6. 交互设计 → 宣查交互体验
7. 可用性   → 检查提示反馈
8. 数据埋点 → 分析数据采集设计
9. 安全设计 → 审查安全机制
```

## 输入处理

### 输入优先级（重要）

```
优先级 1: 已有设计文档（设计稿/原型/PRD/产品文档）
优先级 2: HTML 文件夹（设计稿导出）
优先级 3: 代码库中的 UI 实现
优先级 4: 用户口述描述
```

### 接收输入

用户可能提供：
- **设计稿/原型**（优先）：Figma/Sketch/蓝湖导出的 HTML、图片
- **PRD/产品文档**：Word/PDF/Markdown 格式的需求文档
- **HTML 文件夹**：设计稿导出的完整文件夹
- **代码库**：用于验证设计实现情况
- **纯文本描述**：用户口述的需求和设计思路

### HTML 文件夹扫描（用于验证和补充）

```
/design/xxx/
├── index.html        # 入口文件
├── pages/            # 页面文件夹
├── components/       # 组件文件夹
└── assets/           # 资源文件
```

### 信息提取

| 信息类型 | 提取位置 | 提取方式 |
|----------|----------|----------|
| 页面标题 | `<title>` 或 `<h1>` | 文本内容 |
| 功能模块 | 页面区块划分 | DOM 结构 |
| 交互元素 | 按钮、链接、表单 | 元素识别 |
| 文案内容 | 文本节点 | 文本提取 |

### 文档读取工具

使用以下脚本读取不同格式的文档：

| 文档类型 | 读取脚本 |
|----------|----------|
| PDF | `scripts/read-pdf.py` |
| Word (.docx) | `scripts/read-word.py` |
| Markdown | 直接使用 Read 工具 |
| HTML | 直接使用 Read 工具 |

**使用示例：**
```bash
# 读取 PDF 文档
python scripts/read-pdf.py design.pdf

# 指定页码范围
python scripts/read-pdf.py design.pdf --pages 1-10

# 输出到文件
python scripts/read-pdf.py design.pdf -o output.md

# 读取 Word 文档
python scripts/read-word.py prd.docx
```

**依赖安装：**
```bash
pip install PyMuPDF python-docx
```

## 输出规范

### ⚠️ 重要：单一文档输出

**所有审查内容必须输出到同一个文档，禁止生成多个文件。**

```
docs/product-design/[人名-编号-项目名]/产品设计审查报告.md
```

**禁止行为**：
- ❌ 每个模块生成独立文件（如 `需求背景.md`、`使用场景.md`）
- ❌ 按 phase 生成多个文档
- ❌ 分散输出到不同路径

**正确行为**：
- ✅ 所有内容追加到 `产品设计审查报告.md` 的对应章节
- ✅ 使用 `##` 标题区分不同模块
- ✅ 每完成一个模块，使用 Edit 工具追加内容到主文档

### 输出路径

所有审查内容输出到**一个结构化的主文档**：

**归档命名规范**： `docs/product-design/[人名-编号-项目名]/产品设计审查报告.md`

**示例**:
- 有编号时: `docs/product-design/张三-PRD001-用户中心/产品设计审查报告.md`
- 无人名/编号时: `docs/product-design/用户中心/产品设计审查报告.md`

```
docs/product-design/[人名-编号-项目名]/产品设计审查报告.md
```

### 文档结构

**核心原则**：每个章节必须包含「当前设计梳理」+「审查意见」两部分

完整文档结构模板见 [output-template.md](references/output-template.md)

**章节结构**：
```
# [项目名] 产品设计审查报告
## 1-16. [各分析模块]
  ### x.1 当前设计梳理
  ### x.2 审查意见
  ### x.3 风险统计
## 附录 A. 待补充项清单
## 附录 B. 设计与实现一致性检查
## 附录 C. 风险汇总
```

**状态标记**：✅ 已设计 / ⚠️ 部分设计 / ❌ 缺失
**风险等级**：🔴 高风险 / 🟡 中风险 / 🟢 低风险

## 执行流程

### 第一步：收集输入材料

询问用户：
1. **设计文档路径**：已有的 PRD/设计文档（Word/PDF/MD）位置
2. **设计稿路径**：设计稿/原型导出文件位置
3. **项目名称**：用于输出文档命名
4. **审查范围**：完整审查还是选择性审查（展示章节清单让用户选择）

### 第二步：读取设计文档

1. 根据文档类型选择合适的读取方式
2. 提取设计文档中的关键信息
3. 记录文档中已有的设计决策

### 第三步：扫描设计稿

1. 分析 HTML/图片等形式的设计稿
2. 发现文档中未提及的设计细节
3. 标记设计与文档不一致的地方

### 第四步：执行分析

根据用户选择，按需读取 references 目录下的模块指南：

| 模块 | 参考文件 | 用途 |
|------|----------|------|
| 需求背景 | [requirements.md](references/requirements.md) | 分析项目背景和需求 |
| 业务逻辑 | [business-logic.md](references/business-logic.md) | 梳理业务规则和流程 |
| 使用场景 | [scenarios.md](references/scenarios.md) | 分析核心使用场景 |
| 新手引导 | [scenarios.md](references/scenarios.md#新手引导) | 分析新用户引导 |
| 边缘场景 | [scenarios.md](references/scenarios.md#边缘场景) | 分析异常情况处理 |
| 上下文场景 | [scenarios.md](references/scenarios.md#上下文场景) | 分析环境适配 |
| 站点地图 | [information-architecture.md](references/information-architecture.md) | 梳理内容结构 |
| 导航结构 | [information-architecture.md](references/information-architecture.md#导航结构) | 分析导航设计 |
| 内容层级 | [information-architecture.md](references/information-architecture.md#内容层级) | 分析信息优先级 |
| 页面设计 | [page-design.md](references/page-design.md) | 扫描页面元素布局 |
| 线框图 | [interaction-wireframe.md](references/interaction-wireframe.md) | 分析页面布局 |
| 用户流程 | [interaction-flow.md](references/interaction-flow.md) | 梳理操作路径 |
| 微交互 | [interaction-micro.md](references/interaction-micro.md) | 分析交互细节 |
| 状态设计 | [interaction-state.md](references/interaction-state.md) | 检查状态覆盖 |
| 提示反馈 | [usability.md](references/usability.md) | 检查提示机制 |
| 数据埋点 | [analytics.md](references/analytics.md) | 分析数据采集设计 |
| 安全设计 | [security.md](references/security.md) | 审查安全机制 |

### 第五步：输出文档

1. 创建 `docs/product-design/[项目名]/` 目录
2. 将所有分析结果写入 `产品设计审查报告.md`
3. 标注 `[待补充]` 的信息
4. 在附录中列出待补充项和一致性检查结果

## 完整审查流程

当用户选择完整审查时，按以下顺序执行，**所有内容写入同一个文档**：

### Phase 1: 需求分析
1. 读取 [requirements.md](references/requirements.md)
2. 执行需求背景分析
3. 写入报告「1. 需求背景」章节
4. 询问用户是否继续

### Phase 2: 业务逻辑
1. 读取 [business-logic.md](references/business-logic.md)
2. 执行业务逻辑分析
3. 写入报告「2. 业务逻辑」章节
4. 询问用户是否继续

### Phase 3: 场景设计
1. 读取 [scenarios.md](references/scenarios.md)
2. 依次执行并写入报告：
   - 使用场景分析 → 「3. 使用场景」
   - 新手引导分析 → 「4. 新手引导」
   - 边缘场景分析 → 「5. 边缘场景」
   - 上下文场景分析 → 「6. 上下文场景」
3. 询问用户是否继续

### Phase 4: 信息架构
1. 读取 [information-architecture.md](references/information-architecture.md)
2. 依次执行并写入报告：
   - 站点地图分析 → 「7. 站点地图」
   - 导航结构分析 → 「7. 导航结构」
   - 内容层级分析 → 「8. 内容层级」
3. 询问用户是否继续

### Phase 5: 页面设计
1. 读取 [page-design.md](references/page-design.md)
2. 执行页面设计分析
3. 写入报告「9. 页面设计」
4. 询问用户是否继续

### Phase 6: 交互设计
1. 依次读取并执行：
   - [interaction-wireframe.md](references/interaction-wireframe.md) → 「10. 线框图分析」
   - [interaction-flow.md](references/interaction-flow.md) → 「11. 用户流程」
   - [interaction-micro.md](references/interaction-micro.md) → 「12. 微交互」
   - [interaction-state.md](references/interaction-state.md) → 「13. 状态设计」
2. 询问用户是否继续

### Phase 7: 可用性
1. 读取 [usability.md](references/usability.md)
2. 执行提示反馈分析
3. 写入报告「14. 提示反馈」
4. 询问用户是否继续

### Phase 8: 数据层
1. 读取 [analytics.md](references/analytics.md)
2. 执行数据埋点分析
3. 写入报告「15. 数据埋点」
4. 询问用户是否继续

### Phase 9: 安全设计
1. 读取 [security.md](references/security.md)
2. 执行安全设计分析
3. 写入报告「16. 安全设计」
4. 询问用户是否继续

### Phase 10: 附录生成
1. 汇总所有待补充项 → 「附录 A」
2. 输出设计与实现一致性检查 → 「附录 B」
3. 输出风险汇总 → 「附录 C」
4. 完成报告
