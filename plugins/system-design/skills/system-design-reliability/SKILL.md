# 系统架构可靠可用性检查

端到端扫描已有系统的架构配置，识别风险点，输出可执行的改进计划。

---

## 核心定位

| 维度 | 说明 |
|------|------|
| **目标** | 识别系统架构层面的可靠性/可用性风险 |
| **输入** | 用户提供的配置信息（按模板填写） |
| **输出** | 单一报告 + 可选 Excel 跟踪清单 |
| **原则** | 有则分析，无则跳过 |

### 与其他 Skill 的区别

| Skill | 定位 | 输入 | 时机 |
|-------|------|------|------|
| system-design-review | 设计文档审查 | 设计文档 | 设计阶段 |
| **system-design-reliability** | 架构健康检查 | 实际配置 | 运行阶段 |
| code-execution-efficiency | 代码性能分析 | 代码 | 开发阶段 |

## 分析维度

### 1. 高可用性

检查系统是否具备故障容忍能力。

| 检查点 | 风险等级 | 说明 |
|--------|----------|------|
| 单点故障 | 🔴 P0 | 核心服务是否有备份 |
| 故障转移 | 🔴 P0 | 故障时能否自动切换 |
| 多活部署 | 🟠 P1 | 是否支持跨机房/跨可用区 |
| 容灾方案 | 🟠 P1 | RTO/RPO 是否明确 |

### 2. 性能瓶颈

检查配置是否合理，是否存在隐患。

| 检查点 | 风险等级 | 说明 |
|--------|----------|------|
| 连接池配置 | 🟠 P1 | 是否过小或过大 |
| 超时设置 | 🟠 P1 | 是否合理，是否级联超时 |
| 缓存策略 | 🟡 P2 | 缓存命中率、穿透/雪崩防护 |
| 慢查询风险 | 🟠 P1 | 数据量增长后的性能隐患 |

### 3. 定时任务

检查定时任务的执行时间和依赖关系。

| 检查点 | 风险等级 | 说明 |
|--------|----------|------|
| 执行时间冲突 | 🟠 P1 | 多任务同时执行抢占资源 |
| 高峰期执行 | 🟠 P1 | 大任务在业务高峰期执行 |
| 依赖链过长 | 🟡 P2 | 上下游依赖复杂，失败影响大 |
| 超时配置 | 🟡 P2 | 任务超时是否合理 |

### 4. 数据可靠性

检查数据的持久化和备份策略。

| 检查点 | 风险等级 | 说明 |
|--------|----------|------|
| 主从复制 | 🔴 P0 | 是否配置，延迟多少 |
| 数据备份 | 🔴 P0 | 备份频率和保留周期 |
| 持久化配置 | 🟠 P1 | Redis AOF/RDB 配置 |
| 数据归档 | 🟡 P2 | 大表是否有归档策略 |

### 5. 安全风险

检查配置层面的安全问题。

| 检查点 | 风险等级 | 说明 |
|--------|----------|------|
| 端口暴露 | 🟠 P1 | 不必要的端口对外开放 |
| 认证机制 | 🔴 P0 | 是否有认证，是否弱密码 |
| 加密传输 | 🟠 P1 | 敏感数据是否加密 |
| 权限控制 | 🟠 P1 | 最小权限原则 |

### 6. 可观测性

检查监控和告警是否完善。

| 检查点 | 风险等级 | 说明 |
|--------|----------|------|
| 核心指标监控 | 🟠 P1 | CPU/内存/磁盘/网络 |
| 业务指标监控 | 🟡 P2 | QPS/延迟/错误率 |
| 告警完善度 | 🟠 P1 | 关键异常是否覆盖 |
| 日志采集 | 🟡 P2 | 是否便于排查问题 |

> 详细检查项见 [check-items.md](references/check-items.md)

#### 风险详情

##### R001: Redis 单点故障

**问题描述**：
生产环境 Redis 为单实例部署，未配置主从复制和哨兵。

**影响分析**：
- Redis 故障 → 缓存不可用 → 请求直达数据库 → 数据库压力骤增 → 可能导致整体服务不可用

**当前配置**：
```
# 提供的 Redis 配置
host: 192.168.1.100
port: 6379
无主从配置
```

**修复建议**：
1. **短期**：配置 Redis 主从 + 哨兵
2. **长期**：迁移到 Redis Cluster

**预期收益**：消除单点故障，可用性从 99% 提升到 99.9%

## 执行要点

1. **按需分析**：只分析用户提供的配置，无则跳过
2. **风险量化**：每个问题都要评估风险等级和影响范围
3. **方案具体**：修复建议要有具体步骤，不要泛泛而谈
4. **计划可执行**：改进计划要有时间线和优先级
5. **可跟踪**：输出 Markdown checklist，方便用户跟踪进度
