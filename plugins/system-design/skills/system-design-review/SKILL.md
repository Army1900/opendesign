# 系统设计审查 Skill

基于**已有设计文档**和**项目代码**进行系统设计梳理和审查，输出**可用于下一步工作的设计文档**。

## 核心原则

### 输出定位

**输出的审查报告是一个"可工作的设计文档"**，不是一次性报告——架构师、开发、测试、运维都能直接基于报告工作，不需要再看原始文档。

### 输入优先级

```
1. 已有设计文档（Word/PDF/MD/设计稿导出）
2. 项目代码实现
3. 配置文件
4. 用户口述描述
```

**不要从零开始设计**，基于现有材料进行梳理和审查。

### 多项目处理

提供多个项目时，**逐个执行，禁止并行**。创建 `系统设计审查计划跟踪.md` 跟踪进度，完成后自动继续下一个。详细流程和示例见 [core-principles.md](references/core-principles.md#多项目处理)。

### 保留原始细节

**当前设计梳理**必须保留原始文档详细内容。禁止概括（"系统有3个服务"）、禁止丢失定义、禁止省略接口细节和配置参数。示例对比见 [core-principles.md](references/core-principles.md#保留原始细节)。

### 审查意见务实原则

**审查意见必须基于实际设计内容给出，禁止凭空编造问题或建议。** 设计文档和代码里有什么就审什么，没有的不审。不确定的标注「[待确认]」而不是猜测。不要为了凑检查项而杜撰不存在的风险。

**如果本次需求/原型/系统没有涉及某个章节的内容，直接写「不涉及」，不要为了输出而强行填充。**

### 前置依赖

**技术栈扫描是所有模块的前置依赖，始终首先执行。** 它识别项目使用的框架和中间件，为后续模块提供技术上下文。

## 章节选择（必须执行）

**开始审查前，一次性展示所有章节让用户选择。** 用户选完后按顺序依次执行所有选中章节，**中途不再询问是否继续**。技术栈扫描始终执行，不在选择范围内。

```
请选择要审查的章节（输入编号，多选用逗号分隔，输入 all 全选）：

| 编号 | 章节 | Phase |
|------|------|-------|
| 1  | 战略设计   | Phase 1 |
| 2  | 服务设计   | Phase 2 |
| 3  | 数据设计   | Phase 3 |
| 4  | API设计    | Phase 4 |
| 5  | 事件设计   | Phase 4 |
| 6  | 横切设计   | Phase 5 |
| 7  | 工作拆分   | Phase 6 |

示例：all | 1,4,5 | 4-6
```

用户选择后，先执行技术栈扫描，再按章节编号顺序依次执行。

## 执行流程

**⚠️ 每个 Phase 必须先读取对应的 reference 文件，然后按其中的输出模板、检查项清单执行分析。禁止跳过 reference 文件直接凭经验输出。**

### 收集输入

询问用户：
1. **设计文档路径**：已有的设计文档（Word/PDF/MD）位置
2. **代码库目录**：项目代码位置
3. **项目名称**：用于输出文档命名
4. **审查范围**：展示章节清单让用户选择

文档读取和代码扫描方法见 [document-reading.md](references/document-reading.md)。

### 创建输出文档

收集完输入后、开始分析前，**立即创建输出文档**：

1. 创建目录：`mkdir -p docs/system-design/[人名-编号-项目名]/`
2. 创建文件：`docs/system-design/[人名-编号-项目名]/系统设计审查报告.md`，写入报告头部（标题、时间、来源、审查范围）
3. 后续每个 Phase 的分析结果，使用 **Edit 工具追加**到该文件的对应章节

### Phase 0: 前置扫描（始终执行）

1. 读取 [prerequisites.md](references/prerequisites.md)
2. 执行技术栈扫描
3. 写入报告「0. 技术栈」章节

### Phase 1: 战略设计

1. 读取 [strategy.md](references/strategy.md)
2. 写入报告「1. 战略设计」章节，用子标题区分：
   - 1.1 系统上下文
   - 1.2 领域模型
   - 1.3 服务拆分

### Phase 2: 服务设计

1. 读取 [service.md](references/service.md)
2. 写入报告「2. 服务设计」章节，用子标题区分：
   - 2.1 服务定义
   - 2.2 服务契约
   - 2.3 数据流程

### Phase 3: 数据设计

1. 读取 [data.md](references/data.md)
2. 写入报告「3. 数据设计」章节，用子标题区分：
   - 3.1 实体设计
   - 3.2 存储设计
   - 3.3 缓存设计

### Phase 4: 通信设计

1. 读取 [communication.md](references/communication.md)
2. 写入报告「4. API设计」和「5. 事件设计」（各自独立章节）

### Phase 5: 横切设计

1. 读取 [crosscutting.md](references/crosscutting.md)
2. 写入报告「6. 横切设计」章节，用子标题区分：
   - 6.1 安全设计
   - 6.2 性能设计
   - 6.3 可用性设计
   - 6.4 数据兼容性

### Phase 6: 工作拆分

1. 读取 [execution.md](references/execution.md)
2. 写入报告「7. 工作拆分」

### 附录生成

1. 汇总所有待补充项 → 「附录 A」
2. 输出设计与代码一致性检查 → 「附录 B」
3. 输出风险汇总 → 「附录 C」

## 输出规范

### 单一文档输出

**所有审查内容必须输出到同一个文档，禁止生成多个文件。**

路径：`docs/system-design/[人名-编号-项目名]/系统设计审查报告.md`

示例：
- 有编号时：`docs/system-design/张三-SDD001-用户中心/系统设计审查报告.md`
- 无人名/编号时：`docs/system-design/用户中心/系统设计审查报告.md`

### 文档结构

每个章节包含「当前设计梳理」+「审查意见」+「风险统计」。完整模板见 [output-template.md](references/output-template.md)

```
# [项目名] 系统设计审查报告
## 0. 技术栈
## 1. 战略设计
  ### 1.1 系统上下文 / 1.2 领域模型 / 1.3 服务拆分
## 2. 服务设计
  ### 2.1 服务定义 / 2.2 服务契约 / 2.3 数据流程
## 3. 数据设计
  ### 3.1 实体设计 / 3.2 存储设计 / 3.3 缓存设计
## 4. API设计
## 5. 事件设计
## 6. 横切设计
  ### 6.1 安全设计 / 6.2 性能设计 / 6.3 可用性设计 / 6.4 数据兼容性
## 7. 工作拆分
## 附录 A. 待补充项清单
## 附录 B. 设计与代码一致性检查
## 附录 C. 风险汇总
```

**状态标记**：✅ 已设计 / ⚠️ 部分设计 / ❌ 缺失
**风险等级**：🔴 高风险 / 🟡 中风险 / 🟢 低风险

## 快捷触发词

| 触发词 | 执行动作 |
|--------|----------|
