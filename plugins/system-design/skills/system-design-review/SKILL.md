# 系统设计审查 Skill

基于**已有设计文档**和**项目代码**进行系统设计梳理和审查，输出**可用于下一步工作的设计文档**。

## ⚠️ 核心原则

### 输出定位

**输出的审查报告是一个"可工作的设计文档"**，不是一次性报告：

| 角色 | 用途 |
|------|------|
| **架构师** | 基于报告进行架构优化，不需要再看原始文档 |
| **开发** | 基于报告进行开发，了解所有服务和接口设计 |
| **测试** | 基于报告编写测试用例，了解所有业务场景 |
| **运维** | 基于报告进行部署规划，了解所有依赖和配置 |

### 输入优先级

```
1. 首先阅读已有的设计文档（Word/PDF/MD/设计稿）
2. 然后结合项目代码实现进行验证和补充
3. 输出：对现有设计的"梳理 + 审查 + 补充建议"
```

**不要从零开始设计**，而是基于现有材料进行梳理和审查。

### ⚠️ 多项目处理规范

当用户提供**多个不同项目**的设计文档时：

**禁止行为**：
- ❌ 为了追求速度，并行分析多个项目
- ❌ 胡乱填充数据，把项目A的内容填到项目B
- ❌ 混淆不同项目的设计内容
- ❌ 在一个报告中合并多个项目

**正确行为**：
1. **创建计划文档**：在当前执行目录创建 `系统设计审查计划跟踪.md`
2. **逐个分析**：按顺序一个一个项目完整分析
3. **更新计划文档**：完成后立即更新计划文档中的状态
4. **自动继续**：无需等待确认，直接开始下一个项目

**计划文档模板**：`系统设计审查计划跟踪.md`（见本目录同名文件）

**示例流程**：
```
用户: 帮我审查这三个项目的系统设计：张三-SDD001-用户中心、订单系统、支付网关

AI: 收到，我将按顺序逐个分析以下项目：

[创建/更新 计划跟踪文档 `系统设计审查计划跟踪.md`]
| 序号 | 项目标识 | 状态 |
|------|----------|------|
| 1 | 张三-SDD001-用户中心 | ⏳ 待开始 |
| 2 | 订单系统 | ⏳ 待开始 |
| 3 | 支付网关 | ⏳ 待开始 |

[更新状态为🔄进行中，开始分析第1个项目「用户中心」...

[分析完成后, 更新计划文档标记✅已完成]
[自动继续第2个项目「订单系统」...
```

### ⚠️ 保留原始细节（重要）

**当前设计梳理**部分必须保留原始文档的详细内容，不要过度概括。

> 📌 **目标**：看报告的人不需要再看原始设计文档，所有关键信息都在报告里。

**禁止行为**：
- ❌ 高度概括，如"系统有3个服务"
- ❌ 丢失具体定义，如"[服务A]"而不是"UserService 用户服务"
- ❌ 忽略接口细节，如"有查询接口"而不是具体描述请求/响应结构
- ❌ 省略配置参数，如"配置了缓存"而不是具体描述缓存策略和TTL

**正确行为**：
- ✅ **逐项记录**：每个服务、接口、实体都要单独记录
- ✅ **保留原文**：服务描述、字段名、配置项保持原文
- ✅ **详细定义**：请求参数、响应结构、字段类型都要描述
- ✅ **记录约束**：非空校验、格式校验、业务规则都要记录

**示例对比**：

| 错误（过度概括） | 正确（保留细节） |
|------------------|------------------|
| 系统有3个微服务 | 服务清单：1) UserService 用户服务（端口8081，负责用户注册/登录/信息管理）；2) OrderService 订单服务（端口8082，负责订单创建/查询/状态流转）；3) PaymentService 支付服务（端口8083，负责支付/退款） |
| API有查询接口 | GET /api/v1/users/{id} - 查询用户详情。请求：路径参数id（必填，Long）。响应：`{"code":0,"data":{"id":1,"name":"张三","phone":"138****1234"}}` |
| 配置了Redis缓存 | 缓存配置：1) 用户信息缓存（Key: `user:info:{id}`，TTL: 30分钟，Hash结构）；2) Session缓存（Key: `session:{token}`，TTL: 2小时，String结构） |

### 前置依赖

**在运行其他模块之前，必须先运行技术栈扫描模块**

技术栈扫描识别项目使用的框架和中间件，是其他模块的前置依赖。

## 审查流程

系统设计审查按以下顺序进行：

```
Phase 0: 前置扫描
└── 技术栈扫描 ← 必须首先运行

Phase 1: 战略层
├── 系统上下文
├── 领域模型
└── 服务拆分

Phase 2: 服务层
├── 服务定义
├── 服务契约
└── 数据流程

Phase 3: 数据层
├── 实体设计
├── 存储设计
└── 缓存设计

Phase 4: 通信层
├── API设计
└── 事件设计

Phase 5: 横切层
├── 安全设计
├── 性能设计
├── 可用性设计
└── 数据兼容性

Phase 6: 执行层
└── 工作拆分
```

## 输出规范

### ⚠️ 重要：单一文档输出

**所有审查内容必须输出到同一个文档，禁止生成多个文件。**

```
docs/system-design/[人名-编号-项目名]/系统设计审查报告.md
```

**禁止行为**：
- ❌ 每个模块生成独立文件（如 `技术栈.md`、`领域模型.md`）
- ❌ 按 Phase 生成多个文档
- ❌ 分散输出到不同路径

**正确行为**：
- ✅ 所有内容追加到 `系统设计审查报告.md` 的对应章节
- ✅ 使用 `##` 标题区分不同模块
- ✅ 每完成一个模块，使用 Edit 工具追加内容到主文档

### 输出路径

所有审查内容输出到**一个结构化的主文档**：

**归档命名规范**: `docs/system-design/[人名-编号-项目名]/系统设计审查报告.md`

**示例**:
- 有编号时: `docs/system-design/张三-SDD001-用户中心/系统设计审查报告.md`
- 无人名/编号时: `docs/system-design/用户中心/系统设计审查报告.md`

```
docs/system-design/[人名-编号-项目名]/系统设计审查报告.md
```

### 状态和风险定义

**状态标记**:
- ✅ 已设计：设计文档和代码实现一致
- ⚠️ 部分设计：有设计但未实现，或实现与设计不一致
- ❌ 缺失：无设计文档且无代码实现

**风险等级**:
- 🔴 高风险：可能导致系统故障或安全问题
- 🟡 中风险：影响系统可维护性或扩展性
- 🟢 低风险：仅影响规范性和一致性

- 每个 chapter 使用 `##` 标题
- 使用表格展示检查项

## 执行流程

### 第一步：收集输入材料

询问用户：
1. **设计文档路径**：已有的设计文档（Word/PDF/MD）位置
2. **代码库目录**：项目代码位置
3. **项目名称**：用于输出文档命名
4. **审查范围**：完整审查还是选择性审查（展示章节清单让用户选择）

### 第二步：读取设计文档

1. 根据文档类型选择合适的读取方式
2. 提取设计文档中的关键信息
3. 记录文档中已有的设计决策

### 第三步：扫描代码库

1. 使用代码扫描验证设计文档中的描述
2. 发现文档中未提及的实现细节
3. 标记设计与代码不一致的地方

1. 创建 `docs/system-design/[项目名]/` 目录
2. 将所有分析结果写入 `系统设计审查报告.md`

4. 在附录中列出待补充项和一致性检查结果

## 完整审查流程

当用户选择完整审查时，按以下顺序执行，**所有内容写入同一个文档**：

### Phase 0: 前置扫描
1. 读取 [prerequisites.md](references/prerequisites.md)
2. 执行技术栈扫描
3. 输出到报告的「1. 技术栈」章节
4. **必须完成后才能继续**

### Phase 1: 战略层
1. 读取 [strategy.md](references/strategy.md)
2. 依次执行并写入报告：
   - 系统上下文分析 → 「2. 系统上下文」
   - 领域模型分析 → 「3. 领域模型」
   - 服务拆分分析 → 「4. 服务拆分」
3. 询问用户是否继续

### Phase 2: 服务层
1. 读取 [service.md](references/service.md)
2. 依次执行并写入报告：
   - 服务定义分析 → 「5. 服务定义」
   - 服务契约分析 → 「6. 服务契约」
   - 数据流程分析 → 「7. 数据流程」
3. 询问用户是否继续

### Phase 3: 数据层
1. 读取 [data.md](references/data.md)
2. 依次执行并写入报告：
   - 实体设计分析 → 「8. 实体设计」
   - 存储设计分析 → 「9. 存储设计」
   - 缓存设计分析 → 「10. 缓存设计」
3. 询问用户是否继续

### Phase 4: 通信层
1. 读取 [communication.md](references/communication.md)
2. 依次执行并写入报告：
   - API设计分析 → 「11. API 设计」
   - 事件设计分析 → 「12. 事件设计」
3. 询问用户是否继续

### Phase 5: 横切层
1. 读取 [crosscutting.md](references/crosscutting.md)
2. 依次执行并写入报告：
   - 安全设计分析 → 「13. 安全设计」
   - 性能设计分析 → 「14. 性能设计」
   - 可用性设计分析 → 「15. 可用性设计」
   - 数据兼容性分析 → 「16. 数据兼容性」
3. 询问用户是否继续

### Phase 6: 执行层
1. 读取 [execution.md](references/execution.md)
2. 执行工作拆分
3. 写入报告「17. 工作拆分」

### Phase 7: 附录生成
1. 汇总所有待补充项 → 「附录 A」
2. 输出设计与代码一致性检查 → 「附录 B」
3. 完成报告

## 快捷触发词

| 触发词 | 执行动作 |
|--------|----------|
| "完整审查" / "全面分析" | 执行所有模块 |
| "快速审查" / "概览" | 只执行技术栈+系统上下文+服务拆分 |
| "架构分析" | 执行 Phase 0-2 |
| "数据架构" | 执行 Phase 3 |
| "技术架构" | 执行 Phase 0-4 |
| "技术栈扫描" | 只执行技术栈扫描 |
