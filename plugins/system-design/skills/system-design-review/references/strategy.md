---

## 输出结构

**每个章节必须包含两部分**：
1. **当前设计梳理**：从设计文档/代码中提取的现有设计内容
2. **审查意见**：基于梳理内容给出的问题、风险、建议

## 检查项清单

### 系统上下文检查项

| 检查项 | 说明 | 风险等级 | 常见问题 |
|--------|------|----------|----------|
| 系统定位 | 核心价值和目标用户明确 | 中 | 定位模糊 |
| 系统目标 | 业务/技术/非功能性目标 | 中 | 目标不清晰 |
| 边界定义 | 系统包含什么/不包含什么 | 中 | 边界不清 |
| 外部依赖 | 依赖系统和数据流向 | 高 | 隐式依赖未识别 |
| 依赖SLA | 外部依赖的可用性要求 | 高 | 未评估依赖风险 |
| 约束识别 | 业务/技术/资源约束 | 中 | 约束未明确 |
| 风险识别 | 已知风险和应对策略 | 高 | 风险未识别 |

### 领域模型检查项

| 检查项 | 说明 | 风险等级 | 常见问题 |
|--------|------|----------|----------|
| 核心域识别 | 区分核心域/支撑域/通用域 | 中 | 核心域不突出 |
| 子域划分 | 子域边界清晰 | 中 | 子域过大或过小 |
| 限界上下文 | 上下文边界明确 | 高 | 边界模糊 |
| 上下文映射 | 上下文间关系清晰 | 高 | 关系混乱 |
| 聚合根识别 | 聚合边界和一致性边界 | 高 | 聚合过大 |
| 领域事件 | 关键业务事件识别 | 中 | 事件遗漏 |
| 通用语言 | 统一术语定义 | 低 | 术语不一致 |

### 服务拆分检查项

| 检查项 | 说明 | 风险等级 | 常见问题 |
|--------|------|----------|----------|
| 拆分原则 | 业务/团队/数据边界 | 中 | 拆分依据不明 |
| 服务粒度 | 服务大小适中 | 高 | 过大或过小 |
| 服务分层 | 网关/BFF/核心/基础 | 中 | 层次不清 |
| 服务职责 | 单一职责原则 | 高 | 职责过多 |
| 依赖关系 | 依赖清晰无循环 | 高 | 循环依赖 |
| 依赖类型 | 同步/异步明确 | 高 | 依赖方式不当 |
| 服务所有权 | 负责团队和代码仓库 | 低 | 所有权不清 |
| 独立部署 | 服务可独立部署 | 高 | 强耦合 |

## 执行要点

1. 读取主报告的「1. 技术栈」章节了解技术框架
2. 分析系统文档/代码，提取战略信息
3. **对照检查项清单逐项审查**
4. 输出带状态和风险的审查结果
5. 标注缺失信息和高风险项
6. 对于不确定的边界，向用户确认
