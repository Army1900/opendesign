## 数据兼容性分析 {#数据兼容性}

分析系统的数据格式兼容性设计，确保新老格式可以平滑过渡。

> **输出位置**：追加到主文档 `## 16. 数据兼容性` 章节

### 前置依赖
- 主报告「1. 技术栈」- 了解数据库/缓存/消息类型
- 主报告「9. 存储设计」- 了解表结构和索引

### 核心问题

数据格式变更时的三个核心问题：
1. 老数据怎么迁移?
2. 新老格式怎么共存?
3. 何时可以删除老格式?

### 分析维度

#### 1. 数据存储维度

##### 数据库 Schema 变更
| 变更类型 | 风险 | 兼容策略 |
|----------|------|----------|
| 新增字段 | 低 | DEFAULT + Nullable |
| 删除字段 | 高 | 废弃标记 → 代码移除 → 删字段 |
| 改名 | 高 | 双写双字段 → 迁移 → 删旧 |
| 改类型 | 中 | 扩大类型 / 转换字段 |

##### 缓存数据变更
| 变更类型 | 问题 | 兼容策略 |
|----------|------|----------|
| Key 结构变化 | 老Key失效 | Key版本化 |
| Value 字段新增 | 反序列化失败 | Optional + 默认值 |

##### 消息数据变更
| 变更类型 | 问题 | 兼容策略 |
|----------|------|----------|
| 消息字段新增 | 老消费者忽略 | Optional + 默认值 |
| 消息格式升级 | 解析失败 | 版本号 + 多版本消费 |

#### 2. 兼容策略模式

##### 双写双读 (Dual Write)
```
写入: 写老格式 + 写新格式
读取: 先读新格式, 失败则读老格式
迁移: 后台任务迁移老数据到新格式
清理: 确认全部迁移后, 删除老格式
```

##### 版本号字段 (Version Field)
```
消息中嵌入版本号:
{
    "version": 2,
    "data": { ... }
}

消费者按版本解析
```

##### 渐进式迁移 (Progressive Migration)
```
阶段1: 新代码可读新老格式
阶段2: 新代码写入新格式
阶段3: 后台任务迁移老数据
阶段4: 验证迁移完成
阶段5: 新代码不再处理老格式
阶段6: 删除老格式数据
```

#### 3. 迁移任务清单
| 编号 | 任务 | SQL/代码 | 状态 |
|------|------|----------|------|
| 1 | 迁移用户数据 | UPDATE ... | 待执行 |

#### 4. 验证脚本
| 编号 | 验证项 | 检查SQL | 通过标准 |
|------|--------|---------|----------|
| 1 | 数据一致性 | SELECT COUNT(...) | = 0 |

#### 5. 回滚预案
| 时间点 | 问题 | 回滚操作 |
|--------|------|----------|
| 加字段后 | 字段设计错误 | DROP COLUMN |
| 双写后 | 性能问题 | 关闭新字段写入 |
