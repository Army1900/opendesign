# 执行层分析：工作拆分

将系统设计拆分为可执行的工作子项，粒度控制在 1-5 人天。

---

## 输出结构

**每个章节必须包含两部分**：
1. **当前设计梳理**：从设计文档中提取的现有工作拆分
2. **审查意见**：基于梳理内容给出的问题、风险、建议

## 方法论

本 skill 结合三种方法：

| 方法论 | 应用 |
|--------|------|
| **WBS (Work Breakdown Structure)** | 层级分解, 可交付物导向 |
| **DDD 限界上下文** | 按领域边界分组 |
| **敏捷 User Story** | 粒度控制, INVEST 原则 |

---

## 拆分原则

### 1. 按领域分组

```
工作项按 DDD 限界上下文分组:
- 用户域
- 订单域
- 商品域
- 支付域
- ...
```

### 2. 粒度控制

| 粒度 | 人天 | 处理方式 |
|------|------|----------|
| 太大 | > 5人天 | 继续拆分 |
| 合适 | 1-5人天 | 保持 |
| 太小 | < 1人天 | 合并到相关项 |

### 3. INVEST 原则

| 原则 | 说明 | 检查项 |
|------|------|--------|
| **I**ndependent | 独立性 | 不依赖其他未完成项 |
| **N**egotiable | 可协商 | 范围可调整 |
| **V**aluable | 有价值 | 对用户/业务有价值 |
| **E**stimable | 可估算 | 能给出合理人天 |
| **S**mall | 足够小 | 1-5人天 |
| **T**estable | 可测试 | 有明确验收标准 |

### 4. 工作项类型

| 类型 | 图标 | 说明 | 典型内容 |
|------|------|------|----------|
| 功能开发 | 🚀 | 业务功能实现 | API、页面、逻辑 |
| 技术基建 | 🔧 | 基础设施搭建 | 框架、中间件、工具 |
| 数据迁移 | 📦 | 数据相关 | 迁移脚本、数据清洗 |
| 测试验收 | ✅ | 测试相关 | 单测、集成测试、E2E |
| 文档编写 | 📝 | 文档相关 | 设计文档、API文档 |

## 拆分步骤

```
1. 读取已有设计文档
2. 按领域分组 (DDD 限界上下文)
3. 每个领域识别工作项:
   - 功能开发项 (API/Service/Repository)
   - 技术基建项 (配置/中间件)
   - 测试验收项
4. 检查粒度 (1-5人天)
5. 标注依赖关系
6. 识别关键路径
7. 规划迭代
8. 输出文档
```
