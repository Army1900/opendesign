# 代码执行效率分析

本 skill 用于系统性地分析代码执行效率问题，从调用链上游到下游全链路排查，识别性能瓶颈并提供优化方案。

### 2. 调用链路分析

#### 2.1 绘制调用链路图

从入口方法开始，逐层追踪到最下游（通常是数据库或外部服务）：

```
┌─────────────────────────────────────────────────────────────┐
│                      调用链路图                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [入口] Controller/API                                       │
│      │                                                      │
│      ▼                                                      │
│  [业务层] Service.method()                                   │
│      │                                                      │
│      ├─► [调用A] 内部方法计算 ◄── 耗时: 50ms                 │
│      │                                                      │
│      ├─► [调用B] 数据库查询 ◄── 耗时: 800ms ⚠️ 慢            │
│      │      │                                               │
│      │      ├─► SQL 1: SELECT ... ◄── 500ms ⚠️              │
│      │      └─► SQL 2: UPDATE ... ◄── 300ms                 │
│      │                                                      │
│      ├─► [调用C] 外部API调用 ◄── 耗时: 200ms                 │
│      │                                                      │
│      └─► [调用D] 缓存操作 ◄── 耗时: 5ms                      │
│                                                             │
│  总耗时: ~1055ms                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 2.2 链路分析要点

- **每一层耗时**：标记超过阈值的部分（通常 >100ms 需关注）
- **调用次数**：是否存在循环调用、N+1 问题
- **同步/异步**：串行执行的可否并行化
- **资源竞争**：锁等待、连接池耗尽

### 3. SQL 性能分析

#### 3.1 常见 SQL 性能问题清单

| 问题类型 | 识别特征 | 风险等级 |
|---------|---------|---------|
| **全表扫描** | EXPLAIN 显示 type=ALL | 🔴 高 |
| **缺失索引** | key=NULL，rows 过大 | 🔴 高 |
| **索引失效** | 使用函数、隐式转换、前缀模糊 | 🔴 高 |
| **回表过多** | Using index condition + 大量回表 | 🟡 中 |
| **临时表** | Using temporary | 🟡 中 |
| **文件排序** | Using filesort + 大数据量 | 🟡 中 |
| **锁等待** | SELECT ... FOR UPDATE 长事务 | 🔴 高 |
| **大事务** | 多 SQL 在同一事务，执行时间长 | 🔴 高 |
| **批量操作** | 单条循环插入/更新 | 🟡 中 |

#### 3.2 SQL 分析模板

```sql
-- 原始 SQL
SELECT * FROM orders WHERE DATE(create_time) = '2024-01-01';

-- 问题分析
-- 1. 使用了函数 DATE()，导致索引失效
-- 2. SELECT * 查询了不需要的字段
-- 3. 可能数据量很大

-- 优化方案
-- 方案1：修改条件避免函数
SELECT id, user_id, amount, status
FROM orders
WHERE create_time >= '2024-01-01 00:00:00'
  AND create_time < '2024-01-02 00:00:00';

-- 方案2：确保有索引
-- CREATE INDEX idx_create_time ON orders(create_time);

-- 方案3：如果数据量大，考虑分页或分区
```

#### 3.3 高并发 SQL 问题

| 场景 | 问题 | 解决方案 |
|-----|------|---------|
| **热点数据更新** | 行锁竞争严重 | 乐观锁、分布式锁、队列异步处理 |
| **大批量写入** | 长事务阻塞 | 分批提交、批量 INSERT |
| **高频查询** | 数据库压力大 | 缓存层、读写分离 |
| **计数器更新** | 锁竞争 | 缓存计数、定时同步 |
| **秒杀扣库存** | 超卖/死锁 | 预扣库存、Redis 原子操作 |

**高并发 SQL 审查清单：**
```markdown
□ 是否有 WHERE 条件？是否走索引？
□ 是否有长事务？（检查事务边界）
□ 是否有锁表操作？（DDL、FOR UPDATE）
□ 是否有批量操作？是否分批？
□ 是否可以先查缓存再查数据库？
□ 写操作是否可以异步化？
□ 是否有热点数据竞争问题？
```

### 4. 接口性能分析

#### 4.1 接口慢的常见原因

| 层级 | 问题 | 排查方法 |
|-----|------|---------|
| **网络** | 带宽不足、延迟高 | ping、traceroute、抓包 |
| **网关** | 限流、排队 | 网关日志、监控指标 |
| **应用** | CPU/内存不足、GC 频繁 | JVM 监控、线程 dump |
| **数据库** | 慢查询、连接池满 | 慢查询日志、连接池监控 |
| **外部依赖** | 第三方服务慢 | 超时配置、熔断器 |

#### 4.2 接口分析输出模板

```markdown
## 接口性能分析报告

### 基本信息
- 接口名称: /api/order/create
- 当前响应时间: 2000ms
- 期望响应时间: 500ms
- 并发量: 100 QPS

### 调用链路
[调用链路图]

### 性能瓶颈
1. 🔴 数据库查询慢（800ms）
   - SQL: SELECT * FROM orders WHERE ...
   - 问题: 缺少索引，全表扫描
   - 影响: 占总耗时 40%

2. 🟡 第三方支付接口慢（500ms）

|-------|-------|---------|---------|

| P1 | 支付接口异步化 | 减少 400ms | 中 |
| P2 | 优化序列化 | 减少 200ms | 低 |
```

### 5. 日志分析

#### 5.1 从日志识别性能问题

**关键日志模式：**

```
# 慢操作日志
[WARN] Slow query detected: SELECT ... took 2345ms

# 超时日志
[ERROR] Timeout waiting for response from payment-service after 5000ms

# 连接池耗尽
[ERROR] Connection pool exhausted, waiting for available connection

# GC 频繁
[INFO] GC pause: 234ms, total: 45 times in last minute

# 锁等待
[WARN] Lock wait timeout: transaction waited 30s for lock
```

#### 5.2 日志分析步骤

1. **收集日志**：获取问题时间段前后的日志
2. **过滤关键信息**：ERROR、WARN、Slow、Timeout 等关键词
3. **统计耗时分布**：找出耗时异常的操作
4. **关联分析**：同一请求 ID 的所有日志串联
5. **定位根因**：从现象追溯到代码位置

### 数据库相关

#### MySQL
- 检查索引：`EXPLAIN` 分析执行计划
- 关注：type、key、rows、Extra 字段
- 慢查询日志：`slow_query_log`
- 锁分析：`SHOW ENGINE INNODB STATUS`

#### PostgreSQL
- 检查索引：`EXPLAIN ANALYZE`
- 关注：Seq Scan、执行时间
- 连接池：pgbouncer 配置
- VACUUM 分析

#### MyBatis / MyBatis-Plus
- 检查 XML 中的 SQL 是否有 `select *`
- 动态 SQL 是否导致索引失效
- 结果集映射是否有 N+1

## 注意事项

1. **先诊断后开药**：不要在没有分析清楚的情况下给出优化建议
2. **数据驱动**：尽量用实际数据（耗时、QPS、数据量）说话
3. **优先级明确**：P0/P1/P2 分清楚，让用户知道先做什么
4. **收益量化**：每个优化方案要说明预期收益
5. **风险评估**：说明优化的潜在风险和回滚方案
6. **验证方法**：告诉用户如何验证优化效果
